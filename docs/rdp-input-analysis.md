# RDP 入力の解析（実測, 2026-06-14）

iPad の「Windows App」RDP クライアントから日本語を入力しながら、`ime-probe`
（素の Win32 ウィンドウ）でキャプチャした記録。

## 所見 1 — ホスト側の IME 変換は発生していない

`probe.jsonl` より:

- すべての文字が、**`vk = 231`（`VK_PACKET`, 0xE7）** の `WM_KEYDOWN` を伴う
  `WM_CHAR` として届く。
- `WM_IME_STARTCOMPOSITION` / `WM_IME_COMPOSITION` /
  `WM_IME_ENDCOMPOSITION` は **一切ない**。
- ローマ字の `WM_CHAR` も、`GCS_COMPSTR` / `GCS_RESULTSTR` も無い。

`VK_PACKET` は `SendInput` + `KEYEVENTF_UNICODE` が生成するものだ。つまり
**iPad 側で日本語変換を済ませ、確定済みの Unicode をホストへ注入している**。
ホスト側の TSF / IMM32 スタックは完全にバイパスされている。

## 所見 2 — 変換は「読みを打つ → Backspace → 打ち直す」という手順

時系列で再構成したストリーム:

```
VK_PACKET  こ ん に ち は     読みが実文字としてそのまま注入される
Backspace x5                  読みを消去
VK_PACKET  今 日 は 仕 事     変換後テキストを注入
Backspace x6
VK_PACKET  も し
...
```

iPad はホストアプリ内に候補ウィンドウを描画できないため、**読みを確定文字として
注入**し、それを **合成 Backspace で消去**してから **変換後テキストを注入**する。

## 所見 3 — なぜメモ帳だけ壊れるのか

- 旧来の Win32 エディットコントロールや大半のアプリは、この「Backspace して
  打ち直す」合成入力を正しく処理する。読みはきれいに消去され、変換結果が
  それを置き換える。（ユーザー報告: 現状メモ帳以外はすべて問題なし。）
- **Windows 11 の新しいメモ帳（v11.2604.x、TSF / RichEdit ベース）** は、この
  合成 backspace-replace を取りこぼす。変換結果が挿入される前に読みが消し切れず、
  **文字が二重化**する。

  例: 入力したい `変換が入るとおかしくなる。` が
  `へんかんがはいるとおか変換入るとおかしくなる。`
  （読みの残骸 ＋ 変換結果）になってしまう。

## KanaBridge への含意

- 当初の前提（TSF テキストサービス / IMM32 を介して `GCS_RESULTSTR` と漏れた
  ローマ字を正規化する）は、**この入力経路には当てはまらない**。正規化すべき
  ホスト側の変換そのものが存在しない。
- 効く手立ては、**注入される `VK_PACKET` + `Backspace` ストリーム**だ。これは
  グローバルな低レベルキーボードフック（`WH_KEYBOARD_LL`）が、イベントが
  メモ帳に届く *前* に捕捉できる。パッケージ版メモ帳のプロセスへ DLL を
  注入する必要もない。
  注意: この RDP 入力では `LLKHF_INJECTED` フラグは **立たない**（通常の
  ハードウェア入力として届く）ため、`VK_PACKET` 自体をマーカーとして使う。

## 対策: `kanafix`

`kanafix` クレートとして実装。設定された対象アプリが前面プロセスの間、合成された
`VK_PACKET` / `Backspace` イベントを抑止し、ワーカースレッドから一定間隔（既定
6 ms）で再注入する。確定バーストを引き伸ばすことで、エディタの TSF が
backspace-replace の競合に負けなくなる。再注入したイベントは `dwExtraInfo` に
印を付け、フィードバックループを防ぐ。
